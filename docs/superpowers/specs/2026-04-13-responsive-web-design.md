# Responsive Web Design

## 目标

在不删减现有 Web 功能、不破坏现有桌面端视觉基调的前提下，把当前站点改造成桌面端和手机 Web 都可用的响应式站点。重点解决顶部导航挤压、首页三栏布局在窄屏下失衡、消息页和个人页在手机上不可操作，以及登录注册页在小屏设备上信息层级混乱的问题。

## 约束

- 保留现有信息架构和主要 UI 风格，不做整站重画。
- 不通过“移动端隐藏功能”来换取适配，核心功能都必须可用。
- 前后端接口不为这次适配额外变化，改造集中在前端布局、交互层级和少量可视文案组织。
- 桌面端不能因为适配手机而退化。
- 远程 Docker 环境需要做真实移动视口验证，而不是只看本地截图。

## 当前问题

### 壳层

- 顶部 `topbar` 仍按桌面三列布局组织，按钮、搜索框、头像和品牌文案在窄屏下容易挤压。
- `mobile-nav` 只是简单浮层，没有和顶部动作区形成互补关系。
- 页面底部给移动端导航预留空间不足，部分页面内容和固定导航互相遮挡。

### 首页

- `hero--home` 与 `page-grid--home` 在平板和手机宽度下只是粗暴改成单列，缺少顺序重排与卡片尺寸控制。
- 左右侧边栏的“热议话题”“校园公告”“推荐路径”在手机端显示层级不合理，占据过多首屏高度。
- 筛选条、话题条、统计卡片和帖子列表没有针对触屏宽度做节奏调整。

### 登录/注册

- 双栏 `auth-layout` 虽然已有断点，但窄屏下卡片和说明区的间距、标题尺寸、表单布局仍偏桌面化。
- 表单按钮和输入区域在手机上不够紧凑，容易出现首屏内容过长。

### 个人页/消息页

- 消息中心仍采用桌面双栏主布局，手机上对话列表和会话内容会互相抢空间。
- 个人页头部信息块、统计信息和操作按钮没有针对小屏做优先级分层。

### 帖子与评论

- 帖子头部、操作区、评论区、回复区在窄屏下缺少换行和按钮尺寸策略。
- 弹窗表单在手机端虽然能显示，但没有统一的宽度、内边距和滚动体验规范。

## 方案比较

### 方案 A：局部补丁式断点修复

做法：
- 只在现有 CSS 上追加几个 `@media` 规则。
- 按页面逐个修补溢出、换行和隐藏问题。

优点：
- 上手快。

缺点：
- 容易继续产生“这个页面好了，另一个页面又垮了”的问题。
- 组件之间没有统一响应式约束，后续功能再加一轮还会失稳。

### 方案 B：响应式壳层重构型

做法：
- 先统一 `AppShell`、`container`、`page-stack`、按钮、卡片、弹窗等全局响应式约束。
- 再针对首页、认证页、消息页、帖子卡片逐块重排布局顺序。

优点：
- 变更集中，扩展性更好。
- 后续再加小程序/App 时，现有 Web 的组件分层会更清晰。
- 更符合你要求的“不要只打补丁”。

缺点：
- 首轮改造会涉及全局样式和多个页面测试。

### 方案 C：拆出独立移动版页面

做法：
- 额外写一套手机端页面或路由。

优点：
- 可以针对移动端单独优化。

缺点：
- 维护成本高。
- 当前项目阶段不适合，后续还会加功能，双份页面会快速失控。

## 选定方案

采用方案 B。

理由：
- 这是当前复杂度和可扩展性之间最平衡的做法。
- 不改后端契约，只清理前端布局与交互层次，风险可控。
- 能为后续扩展小程序/App 时的组件拆分提供更干净的边界。

## 设计

### 1. 全局壳层

- `AppShell` 顶部改为“品牌区 + 动作区 + 移动端补充导航”三层策略。
- 桌面端保留完整顶部导航、搜索、通知、发布、个人入口。
- 平板端压缩品牌副标题、搜索框宽度和按钮文案，优先保证主要动作可点击。
- 手机端顶部只保留品牌、搜索入口、通知入口、发布入口和个人入口中的高频项，主导航下沉到 `mobile-nav`。
- 页面底部统一为固定移动导航预留安全空间，避免被遮挡。

### 2. 首页

- `hero--home` 在桌面保留双列，在平板改成上下结构，但统计卡片仍保持横向网格；在手机改成单列紧凑卡片。
- `page-grid--home` 调整为：
  - 桌面：左右侧栏 + 主内容三栏。
  - 平板：主内容优先，其余模块下沉。
  - 手机：主内容、热议话题、公告、推荐路径按业务优先级单列排序。
- 筛选条和话题条使用横向滚动与自动换行结合，避免按钮挤爆一行。

### 3. 认证页

- 保留双栏视觉，但在 `720px` 以下改成内容优先的单栏流程。
- 表单间距、按钮尺寸、标题字号按手机视口收缩。
- 说明区不删除，只在窄屏下后置。

### 4. 个人页与消息页

- 消息中心在手机端切换为“列表视图 / 会话视图”分层显示，而不是强行双栏并排。
- 会话打开后保留返回入口，消息输入区固定在底部可操作区域。
- 个人页头部信息和操作按钮按上下结构排布，避免横向挤压。

### 5. 帖子、评论与弹窗

- `PostCard` 的作者区、操作区、评论区按钮统一支持换行。
- 评论 composer、回复框、删除/举报/私信按钮在窄屏下改成纵向或自动折行布局。
- 弹窗最大高度受视口限制，内容区可滚动，底部操作区固定可见。

## 数据与接口影响

- 无新增后端接口。
- 只允许前端在展示顺序和布局层做调整，不改变现有业务含义。

## 测试策略

- 为 `AppShell` 增加移动端导航与未读入口可见性测试。
- 为首页增加核心区块在移动布局下仍存在的渲染测试。
- 为消息页增加移动端视图切换测试。
- 运行现有前端测试与构建。
- 使用真实远程域名和移动视口做浏览器 smoke，验证登录、首页、帖子流、消息页可操作。

## 风险与控制

- 风险：全局样式改动可能影响桌面端现有布局。
  控制：优先用新增 class 和分层断点，不直接推翻桌面样式。
- 风险：消息页移动端切换若处理不当，会和现有自动选中首个会话逻辑冲突。
  控制：先补测试，再改布局逻辑。
- 风险：远端缓存旧前端资源导致验证结果失真。
  控制：部署后在真实域名重新加载并用移动视口回归。
