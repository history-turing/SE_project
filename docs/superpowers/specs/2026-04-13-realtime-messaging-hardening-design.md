# Realtime Messaging Hardening Design

## 目标

在保证线上页面真实可信、前后端契约一致的前提下，完成以下三项改造：

- 清理运行时硬编码，移除生产地址、演示帖子、演示会话、演示统计对真实页面的污染
- 将现有私信能力升级为真正的即时通信，在线用户无需刷新即可看到新消息、撤回和未读变化
- 为顶部消息入口增加私信未读红点和未读数聚合，并为评论、回复、点赞、系统通知预留统一扩展位

本次设计的第一优先级不是“少改前端”，而是“前后端、状态层和最终 UI 保持一致”。如果现有页面结构会持续放大不一致问题，则允许同步调整前端页面。

## 当前现状

### 已具备能力

- 后端已拆出独立 `message-service`
- `message-service` 已接入 `RabbitMQ + WebSocket + Redis session registry`
- Gateway 已支持 `/ws/messages` 路由和 `X-User-Id` 透传
- 私信发送、撤回、已读等 HTTP API 已经存在
- 会话列表和会话详情接口已能返回未读数

### 当前问题

- 前端没有建立任何 WebSocket 连接，也没有消费实时消息事件
- 私信页面主要依赖 HTTP 请求和本地拼接状态，新消息通常要刷新后才能完整体现
- 顶部铃铛只是静态按钮，没有未读聚合和红点逻辑
- `AppContext` 仍以 `siteData.ts` 中的演示数据作为运行时兜底，接口失败时会退回假数据
- 生产地址和允许来源仍存在运行时硬编码
- 前端类型、前端状态、后端 DTO 和最终页面展示之间缺少统一契约，容易出现“后端返回一套，前端拼一套，页面又显示另一套”
- 匿名帖子作者目前仍可能通过私信入口暴露真实用户名，违背匿名发布的安全边界

## 设计原则

- 真实数据优先：运行时页面不再依赖演示业务数据填充真实区域
- 一致性优先：以后端接口返回、前端状态和最终页面展示三者一致为最终标准
- 事件驱动优先：消息实时性依赖后端事件推送，不依赖前端轮询兜底
- 通知聚合优先：虽然本期只落地私信红点，但状态模型按统一通知中心设计
- UI 稳定优先：默认尽量不改页面骨架，但一旦旧页面结构妨碍一致性，就允许同步调整前端
- 可恢复优先：WebSocket 断开后允许自动重连，并通过 HTTP 重拉恢复一致性

## 方案选型

### 方案 A：HTTP 轮询

做法：

- 前端定时轮询会话列表和当前会话详情
- 顶部铃铛红点由轮询结果计算

优点：

- 前端改动较小

缺点：

- 体验不是真正的即时通信
- 请求量更高
- 未读数和消息流一致性差
- 不符合当前项目已经引入 WebSocket、MQ 的演进方向

### 方案 B：HTTP 写入 + WebSocket 下行事件

做法：

- 发消息、撤回、标记已读继续走 HTTP API
- 后端通过 RabbitMQ 事件驱动 WebSocket 向在线用户推送消息变更
- 前端统一消费消息事件，更新会话、消息流和顶部通知聚合

优点：

- 复杂度可控
- 充分利用已有基础设施
- 体验接近成熟社区产品
- 便于后续扩展评论、点赞、系统通知

缺点：

- 需要补齐前端实时客户端和更丰富的事件 payload

### 选定方案

采用方案 B。

原因：

- 它是当前项目状态下收益最高、结构最稳的方案
- 不需要现在就额外拆独立通知服务，但能为后续通知中心平滑扩展
- 可以在必要时调整前端局部结构，同时保持整体 UI 风格不跑偏

## 运行时硬编码治理

### 允许保留的静态内容

以下内容可以继续保留在前端静态文件中：

- 导航项
- 图标映射
- 发帖话题枚举
- 纯展示类文案和空状态文案

### 必须移除的运行时假数据

以下内容不得再作为真实页面的运行时兜底：

- 首页帖子列表
- 校友圈帖子列表
- 我的帖子列表
- 私信会话列表
- 私信消息详情
- 真实统计数字
- 真实通知列表
- 真实用户资料统计值

### 前端运行时策略

- `AppContext` 初始值改为“空集合 + loading 状态”
- 页面在数据未返回时显示 loading / skeleton
- 请求失败时显示空状态或错误提示，不再悄悄回退到假帖子和假私信
- 前端页面只消费真实 DTO 或其唯一映射层，不允许页面组件自己兜底拼假数据

### 后端配置策略

- `WebCorsProperties` 的允许来源改为配置驱动
- 本地开发默认值允许 `localhost:5173`、`127.0.0.1:5173`、`localhost:4173`
- 生产域名或 IP 通过环境变量注入，不在 Java 类中写死
- 测试中如果需要断言生产来源，改为基于显式测试配置，而不是硬编码生产服务器 IP

## 一致性治理

### 统一契约

- 私信会话摘要、会话详情、消息详情、通知聚合都必须有单一的前后端契约
- HTTP DTO 与 WebSocket DTO 对同一业务语义必须保持字段名称和含义一致
- 不允许 `api.ts`、`AppContext.tsx`、页面组件分别维护半兼容结构

### 映射层

- 建立集中映射层，专门处理“后端 DTO -> 前端 ViewModel”
- 页面组件只消费 ViewModel，不在组件内部二次拼装业务字段
- 任何字段名或语义变化，只允许在映射层集中调整

### UI 一致性要求

- 同一条业务数据必须同时验证接口返回、前端状态和最终 UI 展示三处一致
- 如果现有页面结构无法稳定表达 `loading / empty / error / reconnecting / active conversation` 等状态，则允许局部改动页面结构
- “为了少改 UI 而保留前后端不一致”是禁止的

## 匿名私信安全设计

### 目标

- 用户从匿名帖子发起私信时，不得看到对方真实用户名、真实 userCode、真实头像来源
- 被私信方在该匿名会话中也不得直接获知发起方真实身份
- 匿名帖子衍生出的私信会话，必须与普通实名私信会话在模型上区分

### 会话模型要求

- 普通实名私信继续使用真实用户关系建立单聊
- 从匿名帖子发起的私信，建立“匿名会话视图”
- 该匿名会话至少要绑定：
  - 来源帖子 `postCode`
  - 发起方真实 `userId`
  - 接收方真实 `userId`
  - 双方匿名展示身份

### 展示身份规则

- 匿名会话中，前端收到的会话摘要和消息详情只允许包含匿名展示名、匿名头像或匿名占位图
- 禁止在匿名会话 DTO 中返回真实 `username`
- 禁止在匿名会话 DTO 中返回可直接反查真实身份的 `userCode`
- 如果业务上必须保留真实用户关系，只允许在服务端内部保存，不允许透传到前端

### 匿名展示建议

- 对匿名帖子作者，统一展示为与帖子一致的匿名身份，例如“匿名树洞作者”
- 对发起私信的一方，也使用匿名会话内的展示身份，例如“匿名同学”
- 匿名身份应在同一匿名会话内保持稳定，但不要求跨帖子复用同一匿名标签

### 入口规则

- 只有当帖子为匿名帖子时，帖子头像/作者区的私信入口才进入匿名私信流程
- 非匿名帖子继续走普通实名私信流程
- 评论作者入口如果来源于匿名帖子语境，也必须遵守同样的匿名保护规则

### 前后端契约要求

- 发起匿名私信时，前端必须显式告诉后端当前入口来自匿名帖子，而不是默认走普通私信接口
- 后端返回的会话类型中应明确区分普通会话与匿名会话，例如：
  - `DIRECT`
  - `ANONYMOUS_POST`
- 前端根据会话类型决定展示真实资料还是匿名资料，不能自行猜测

### 安全约束

- 匿名会话中禁止展示可跳转到真实用户主页的链接
- 匿名会话中禁止复用实名私信已有的真实头像、真实昵称、副标题
- 后台审计仍可保留真实映射关系，但仅超级管理员在审计场景下可见

### 测试要求

- 从匿名帖子发起私信时，前端页面不显示真实用户名
- 匿名会话详情接口不返回真实用户名和真实 userCode
- 匿名会话的实时事件 payload 不返回真实身份字段
- 非匿名帖子发起私信时，仍保持普通实名私信行为

## 实时私信设计

### 整体流程

1. 用户登录后，前端建立 `/ws/messages?token=...` 连接
2. 用户发送消息时，前端仍调用 HTTP 发送接口
3. 后端消息落库后发布消息事件到 RabbitMQ
4. `message-service` 消费事件并向目标在线用户 WebSocket 推送
5. 前端收到事件后更新会话列表、消息流和未读聚合
6. 断线重连后，前端通过重拉会话列表和当前会话详情修正状态

### 前端连接策略

- 登录成功后自动建立连接
- 登出时主动关闭连接
- token 失效或服务拒绝握手时停止重试并回到登录态
- 普通网络波动时使用指数退避自动重连
- 重连成功后自动重拉当前所需数据，修复可能丢失的事件

### 前端事件消费策略

前端至少消费以下事件：

- `message.created`
- `message.recalled`
- `conversation.read`
- `notification.unread.changed`

处理原则：

- 当前打开会话收到 `message.created` 时，直接插入消息流
- 非当前会话收到 `message.created` 时，只更新会话摘要、排序和未读数
- 收到 `message.recalled` 时，替换对应消息展示状态，并同步会话最后一条消息
- 收到已读或未读聚合变更事件时，同步顶部铃铛状态

### 后端事件 payload 调整

当前 `MessageEventDto` 只有：

- `type`
- `conversationCode`
- `messageCode`
- `senderUserId`
- `targetUserIds`

这不足以直接驱动前端 UI，因此本次需要扩成完整实时事件结构，至少包含：

- 事件类型
- 会话编码
- 目标用户 ID 列表
- 当前消息摘要
- 当前会话摘要
- 当前会话未读数
- 当前用户总私信未读数

建议统一为：

- `MessageRealtimeEvent`
  - `type`
  - `conversation`
  - `message`
  - `recipientStates`

其中：

- `conversation` 用于更新会话列表项
- `message` 用于更新当前消息流
- `recipientStates` 用于按用户更新未读聚合

### 已读策略

- 用户进入会话详情后，前端仍调用已读 HTTP 接口
- 后端完成已读落库后，向当前用户推送 `conversation.read` 或统一未读变更事件
- 前端收到事件后同步清零当前会话和顶部总未读

### 撤回策略

- 撤回继续通过 HTTP API 触发
- 后端更新消息状态后发送 `message.recalled`
- 前端不再依赖刷新接口才能看到撤回结果

## 通知入口设计

### 范围

本期通知入口只落地私信提醒，但状态模型按统一通知聚合设计。

### 前端通知模型

新增统一通知摘要模型：

- `messagesUnread`
- `interactionsUnread`
- `systemUnread`
- `totalUnread`
- `hasUnread`

本期只有 `messagesUnread` 有真实数据，其余字段先固定为 `0`。

### 顶部铃铛交互

- 顶部铃铛显示红点或数字徽标
- 有未读时显示聚合数，没有未读时只显示普通图标
- 点击铃铛直接进入 `/profile?tab=messages`
- 暂不新增复杂下拉面板，避免扩大 UI 改动范围

### 后续扩展

后续评论、点赞、回复、公告只需要接入同一个通知聚合状态，不需要重做顶部入口结构。

## 前端模块调整

### `AuthContext`

- 登录成功后通知实时消息客户端启动
- 登出时销毁连接和通知状态
- 保留当前认证模型，不新增第二套登录态

### `AppContext`

- 移除运行时假会话和假消息兜底
- 持有真实会话列表、当前会话详情、通知聚合和连接状态
- 提供接收实时事件后的状态更新函数
- 将“接口 DTO -> ViewModel”的转换集中到单一映射层

### 新增前端实时客户端

建议新增独立模块：

- `frontend/src/realtime/messageRealtimeClient.ts`

职责：

- 建立 WebSocket 连接
- 管理重连
- 解析消息事件
- 向 React 状态层分发事件

### `AppShell`

- 顶部铃铛接入未读聚合状态
- 不改变整体布局方向，但允许局部新增红点/数字徽标

### `ProfilePage`

- 消息中心继续作为主私信页面
- 会话列表和消息流由 HTTP 初始加载 + WebSocket 增量更新驱动
- 如果现有布局无法稳定表达多状态，允许局部调整页面结构

## 后端模块调整

### `message-service`

- 扩展实时事件 DTO
- 在发送、撤回、已读后构造完整事件
- 统一推送给在线用户
- 提供“按用户查询总私信未读数”的服务方法
- 保证 HTTP 接口与实时事件中的会话摘要、消息摘要、未读聚合语义一致

### `gateway`

- 继续使用现有 `/ws/messages` 路由
- 不新增第二条消息 WebSocket 路径

### `server`

- 调整 CORS 配置来源
- 清理与生产地址耦合的硬编码默认值

## 错误处理

- WebSocket 连接失败时，页面不阻塞 HTTP 会话浏览
- 前端可以保留轻量连接状态，但不要求大幅新增告警 UI
- 重连成功后自动重拉会话列表和当前会话详情，修正错过的事件
- 如果事件解析失败，记录错误并保持当前页面可继续使用
- 当后端返回结构和页面所需结构不一致时，优先修正契约或统一映射层，不允许通过散落在页面中的临时拼接规避问题

## 测试设计

### 前端测试

- 建立 WebSocket 后收到 `message.created`，当前会话消息流即时追加
- 收到其他会话消息时，会话列表未读数和排序即时变化
- 收到 `message.recalled` 时，消息文本切换到撤回态
- 顶部铃铛未读数随事件变化
- 登出后关闭 WebSocket

### 后端测试

- 发送消息后生成完整实时事件
- 撤回消息后生成完整实时事件
- 标记已读后生成未读数变化事件
- 总私信未读数统计逻辑正确

### 集成验证

- 两个账号在线互发消息，接收方无需刷新即可看到新消息
- 接收方未打开消息中心时，顶部铃铛出现红点
- 接收方进入会话后，当前会话未读和顶部总未读同步减少
- 撤回消息后双方页面同步更新
- 同一条业务数据在后端接口返回、前端状态和最终页面展示三处保持一致

### 远程验证

最终验收以远程 Docker 部署环境为准：

1. 推送代码并完成 GitHub Actions 部署
2. 在远程环境检查 `gateway`、`message-service`、`rabbitmq`、`redis` 容器状态
3. 实际打开线上页面验证实时消息和铃铛红点

## 风险与控制

### 风险：前端状态改造后页面空白

控制：

- 页面显式区分 loading / empty / error 三态
- 不再使用假数据兜底，但保证失败时页面仍可渲染

### 风险：WebSocket 连接建立但事件信息不够

控制：

- 一次性补齐后端事件 payload
- 不采用“收到事件后再额外补打一堆接口”的补丁式方案

### 风险：未读数出现前后端不一致

控制：

- MySQL 为事实来源
- WebSocket 做增量刷新
- 重连成功后再用 HTTP 重拉做最终一致性修正

## 非目标

本期不包含以下内容：

- 评论、点赞、回复通知的完整落地
- 通知下拉面板和通知分页列表
- 图片私信
- 表情包资源管理
- 群聊
- 移动端推送

这些能力后续都应在本次通知聚合和实时事件结构基础上扩展，而不是重新设计一套消息入口。
