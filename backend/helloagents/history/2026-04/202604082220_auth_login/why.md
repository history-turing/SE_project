# 变更提案: 武大教育邮箱注册与账号登录

## 需求背景
当前项目已经具备基础前端界面和页面聚合后端，但后端仍使用固定演示用户，缺少真实账号体系。用户希望新增一套最小可用的认证流程，要求首次注册必须使用武汉大学教育邮箱接收验证码并完成校验，注册成功后再使用用户名和密码登录。

## 产品分析

### 目标用户与场景
- 目标用户: 武汉大学在校生或拥有武大教育邮箱的合法用户。
- 首次使用场景: 用户输入 `@whu.edu.cn` 邮箱，接收验证码，校验通过后填写用户名和密码完成注册。
- 日常使用场景: 已注册用户使用用户名和密码登录，登录成功后访问树洞页面、发帖、互动和查看个人中心。

### 价值主张与成功指标
- 价值主张: 把当前“演示用户模式”升级为“真实用户模式”，使帖子、收藏、关注、消息与账号绑定。
- 成功指标:
  - 非武大教育邮箱无法发起注册验证码。
  - 验证码通过后才能完成注册。
  - 用户名 + 密码可以正常登录。
  - 登录后业务接口不再依赖固定用户，而是绑定当前登录账号。

### 人文关怀
- 仅在首次注册时校验教育邮箱，避免用户每次登录都依赖邮箱收信。
- 验证码设置有效期和发送频率限制，减少邮箱骚扰和滥用风险。
- 密码不明文保存，保护账号安全。

## 变更内容
1. 新增认证相关数据表与建表脚本，包括账号凭证、邮箱验证码和登录会话相关支持结构。
2. 新增邮箱验证码发送、注册、登录、查询当前用户、退出登录接口。
3. 引入基于 Bearer Token 的登录态，替换固定 `DemoUserContext`。
4. 新增邮件发送配置与 Redis 验证码/会话存储逻辑。
5. 前端新增登录页、注册页、路由守卫和 token 自动注入。

## 影响范围
- 模块: `domain`、`infra`、`server`、`frontend`
- API: 新增 `/api/v1/auth/*`，现有业务接口改为依赖真实登录态
- 数据: MySQL 新增认证表；Redis 新增验证码和会话 key
- 部署: 需要补充 SMTP、MySQL、Redis 配置

## 核心场景

### 需求1: 教育邮箱注册
用户首次注册时：
- 只能填写 `@whu.edu.cn` 教育邮箱。
- 系统向该邮箱发送 6 位验证码。
- 验证码校验成功后，用户才能设置用户名和密码并完成注册。

### 需求2: 用户名密码登录
用户完成注册后：
- 使用用户名和密码登录。
- 登录成功后服务端签发 token，前端后续请求自动携带。

### 需求3: 登录态驱动业务
用户登录后访问首页、个人页、发帖、互动时：
- 接口根据 token 识别用户。
- 发帖人与互动记录写入当前真实用户，而不是固定演示账号。

## 风险评估
- 风险1: 邮件服务依赖 SMTP 配置，若未配置将无法真实发信。
  - 缓解: 提供清晰配置项与文档，接口错误信息明确。
- 风险2: 现有种子数据强依赖 `user_id = 1`。
  - 缓解: 保留演示用户数据作为初始账号资料，不再把业务逻辑写死到固定用户。
- 风险3: 登录态改造会影响所有现有业务接口。
  - 缓解: 统一抽取认证上下文与拦截器，减少散落改动。
